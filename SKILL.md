# OTA 酒店主页质检报告 · Skill

## 这个 skill 做什么

给定任意一家酒店，以及一个或多个 OTA 平台（携程 / 美团 / 大众点评 / 飞猪），输出一份 **10 页左右的 A4 付费客户级专业质检报告（PDF）**，覆盖：

- 平台表达完整度（档次标识、评分、榜单、标签、图片、简介）
- 相对位置（同商圈同档次竞品的横向对比）
- 转化漏斗诊断（曝光 → 点击 → 浏览 → 下单四段漏水点）
- 叙事缺口（产品有的东西主页是否讲清楚了）
- 平台匹配度（这家酒店的产品和所选平台流量性格是否匹配）
- 杠杆动作（按时间窗分档的可执行建议）

## 核心设计哲学（三条底线，每份报告都要守）

这份 skill 不是"抓数据套模板"的机器。它有三条底线，任何一条破掉这份报告就不合格：

1. **没有事实就不写判断**。每一个结论都必须能追到第一层（可观察层）的某个具体数字或事实。第二层不允许出现"可能""大概""估计"这种弱判断——要判断就给依据，给不出就删。
2. **不说套话、不写空话**。"加强品牌建设""优化用户体验""提升运营质量"这种句子永远不准出现。每条建议必须是**可执行、有对象、有时间窗、有预期效果**的。
3. **事实错误是致命伤**。付费客户会一眼识别出"钻级≠星级"、"黑珍珠是餐饮榜不是酒店榜"、"美团酒店的钻石是会员不是档次"这种基本错误。所有平台术语**必须严格遵照 `references/platform_facts.md`**。

## 触发后的工作流

### Step 0：明确分析粒度 + 所选平台（必须先问）

如果用户没明确说，**必问一句**，让用户回答粒度和平台两件事：

> "需要我做单平台深度分析还是多平台横向对比？另外要看哪几个平台？（携程 / 美团 / 大众点评 / 飞猪，可多选）"

根据回答走两条不同分支：

- **单平台深度模式**（默认）：按一个平台抓全量数据 + 竞品对比，输出单平台质检报告
- **多平台对比模式**：按用户选的 2-4 个平台分别抓核心字段，重点放在"同一家酒店在不同平台的表达差异 + 流量分流问题"，输出多平台对比报告

**记录用户选择的平台清单**——这是后续 Step 0.5 平台可用性预检、Step 2 抓数据、Step 3 写报告的基础。

### Step 1：核实用户想分析的酒店（消歧义 + ID 获取）

**场景 A：用户给了酒店全名且无歧义**（"上海外滩茂悦大酒店""广州南沙花园酒店"）→ 直接进 Step 0.5

**场景 B：用户给了酒店主页链接或 hotel ID**（最省事）→ 直接进 Step 0.5

各平台的 hotel ID 识别规则（通用：链接中的那串数字就是 ID）：
- **携程**：主页链接形如 `hotels.ctrip.com/hotel/<数字ID>.html`，末尾 `.html` 前的那串数字就是酒店 ID
- **美团**：主页链接形如 `hotel.meituan.com/<POI_ID>/` 或 `i.meituan.com/awp/h5/hotel/<POI_ID>/`
- **大众点评**：主页链接形如 `dianping.com/shop/<SHOP_ID>`（酒店和餐厅都走 shop 路径）
- **飞猪**：主页链接中找 `shid=<数字>` 或 `itemId=<数字>` 参数

**统一的提取规则告诉用户**：复制主页链接整条给过来即可，Claude 自己从中提取 ID；或者告诉用户"链接中那串 6-10 位的数字通常就是酒店 ID"。

**注意**：移动端链接（m.ctrip.com / i.meituan.com / m.dianping.com）和 PC 端链接中的 ID **通常相同**，两者都可以用来定位同一家酒店。如果用户给的是移动端链接，用里面的 ID 照样可以 `web_fetch` PC 端主页。

**场景 C：酒店名有歧义**（"花园酒店"可能指广州花园酒店、南沙花园酒店、北京花园饭店等）→ **必须先追问消歧义，不自作主张**

追问时给用户三条消歧义路径（让用户选最方便的一条）：

> "酒店名可能有多个匹配，帮我确认一下：
> 1. 如果你知道是哪个城市/商圈，告诉我城市+位置（如"广州花园酒店，越秀区"）
> 2. 如果你有该酒店的任一平台主页链接，直接贴过来（链接末尾的数字就是 hotel ID）
> 3. 或者我列出几家候选让你选"

**场景 D：酒店搜不到**（可能停业、搬迁、改名）→ 告知用户，提供已知信息让用户确认，**不硬造数据**

### Step 0.5：平台可用性预检（关键，不可跳过）

用户可能选了 4 个平台，但酒店实际只在其中 2-3 个上架。**这一步必须在深度抓取之前做完，否则会写出有编造数据的报告**。

**操作**：对用户选的每个平台，用 web_search 做一次快速验证：

```
web_search("<酒店全名> 携程")
web_search("<酒店全名> 美团")
web_search("<酒店全名> 大众点评")
web_search("<酒店全名> 飞猪")
```

根据验证结果把平台分三类：

| 状态 | 判断依据 | 后续处理 |
|---|---|---|
| **已上架** | 搜到该平台的主页链接 | 进 Step 2 抓深度数据 |
| **未上架** | 搜索结果无该平台主页 / 搜到但显示"未入驻""已下架" | 报告里如实标注，不编数据 |
| **状态异常** | 主页显示"暂停营业""搬迁中""重新装修" | 报告里如实标注状态 |

**决策分支**：

- **全部选中平台都已上架** → 进 Step 2，正常流程
- **部分已上架（比如 4 选 3）** → **先告诉用户实际情况**："你选的 4 个平台里，该酒店在 XX 平台没有上架主页，本报告将基于另外 3 个平台做。" 得到用户确认后再进 Step 2
- **只有 1 个平台已上架但用户选了对比模式** → 告知用户"只有 XX 平台可查，多平台对比不成立，要不要改成单平台深度分析？"
- **所选平台全部都没上架或状态异常** → **不写报告**，告诉用户："该酒店在你选的所有平台都未上架或状态异常，无法分析。可能的情况：① 酒店只上架了其他平台 ② 酒店已停业 ③ 酒店名有偏差。请确认一下。"

**报告里如实标注的写法**：

报告头部"数据口径"部分必须诚实交代：

```markdown
- **数据口径**：
  - 携程 PC 端 + 移动端主页（hotel ID: xxxxx）✓ 已上架
  - 美团酒店主页（shop ID: xxxxx）✓ 已上架
  - 大众点评（shop ID: xxxxx）✓ 已上架
  - 飞猪 ✗ **未上架**，本报告不涉及该平台分析
```

多平台对比章节里，涉及"未上架"平台时**单独列一小节说明**。写法参考 `references/report_template.md` 的多平台对比版。典型写法示范：

> **注：关于飞猪平台**
> 该酒店未在飞猪上架主页。对于一家国际品牌连锁酒店而言，未接入飞猪意味着失去了阿里生态（5300 万 88VIP 用户 + 芝麻信用住 + 会员通直升通道）的完整入口。这不是主页运营问题，而是酒店/集团的渠道战略选择——是否上架飞猪是集团层面需要判断的问题，不在单店主页质检范围。

**严禁做的事**：

- ❌ 某平台没搜到就编造数据凑版面
- ❌ 某平台没搜到就默认"不重要"略过不提
- ❌ 报告里假装 4 个平台都做了分析

### Step 2：按所选平台抓数据（关键步骤，不可省略）

**绝对不允许凭印象写，必须 web_search + web_fetch 拿实际数据**。

读对应的平台差异文件来获取抓取指令和平台独有字段：

- 携程 → 读 `platforms/ctrip.md`
- 美团 → 读 `platforms/meituan.md`
- 大众点评 → 读 `platforms/dianping.md`
- 飞猪 → 读 `platforms/fliggy.md`

**必须拿到的通用数据清单**（四平台共有，缺一不可）：

- [ ] 酒店全名、开业年份、最近装修年份、客房总数
- [ ] 平台档次标识（按平台规则，如携程需要"钻级+国标星级+平台档次"三轨标注）
- [ ] 总分评分 + 所有可见的子项评分
- [ ] 点评总数 + 差评占比（能推算就推算）
- [ ] 至少 5 条具体点评原文（好评 + 差评都要，含用户原字）
- [ ] 一级页面展示图片数 + 相册总图片数（**两个都要**，不能混淆）
- [ ] 酒店简介原文（至少前 3 句）
- [ ] 酒店特色标签列表
- [ ] 设施清单（分类：公区 / 客房 / 餐饮 / 会议 / 亲子 / 安保）
- [ ] 地理位置参数（最近地铁/机场/火车站距离，核心景点/商圈距离）
- [ ] 至少 2 家同商圈同档次竞品的评分、点评数、档次
- [ ] 平台独家信号（见 `platforms/<平台>.md`）

如果上述清单有 3 项以上搜不到，**进入 Step 2.5 做降级决策**。

### Step 2.5：数据质量门控

参见 `references/data_quality_gate.md`。简要：

- **数据充足**（≥12 分） → 标准版 7 章模板
- **数据中等**（6-11 分） → 精简版 5 章，明确标注"样本局限"
- **数据严重不足**（<6 分） → 不出完整报告，改为 1-2 页对话速评

**绝对不为了凑版面写"建议加强品牌定位"这类空话。**

### Step 3：按三层漏斗框架写 Markdown 报告

报告必须严格遵循"**可观察 → 诊断 → 杠杆**"三层结构。

- 读 `references/analysis_framework.md` 学清楚三层各自允许出现什么、禁止出现什么
- 按 `references/report_template.md` 的章节骨架写
- 抽取平台独有字段和术语时，严格对照 `references/platform_facts.md`，避免事实错误

文件保存位置：`/home/claude/<酒店名>_<平台名>_质检报告.md`（单平台）
或 `/home/claude/<酒店名>_多平台对比_质检报告.md`（对比模式）

### Step 4：Markdown → PDF

用 skill 自带的排版脚本：

```bash
python <skill目录>/scripts/make_pdf.py <md文件路径>
```

脚本会自动处理：A4 精排、页眉页脚（标题从 md 第一个 `#` 提取）、**日期自动取当天**、中文衬线字体、表格斑马纹、blockquote 强调框。

**依赖**：

```bash
pip install markdown weasyprint --break-system-packages
# 系统字体：Ubuntu 需 fonts-noto-cjk（通常已有）
```

### Step 5：交付

用 `present_files` 把 PDF 交给用户。不追问集团战略报告（本 skill 只做单店）。

自然收尾即可，如果用户后续有问题再展开。

## 写作红线（严格遵守）

### 第一层（可观察）禁用清单

第一层只陈列客观事实，**不允许出现**：

- 评价词："好""差""不错""优秀""一般"
- 判断词："建议""应该""需要""问题是"
- 推理副词："由此看来""可以推断""显然"

违反就是越层。结论归结论、事实归事实。

### 第二层（诊断）禁用清单

第二层要下判断，**不允许出现**：

- 软弱词："可能""大概""估计""或许""似乎"
- 没有证据的断言：每条判断都必须在同一段里引用第一层的**具体数字或事实**
- 循环论证：A 因为 B、B 因为 A 的结构

### 第三层（杠杆）禁用清单

第三层写建议，**不允许出现**：

- 空话："优化""加强""提升""改善""推进""落实""建立""完善"（这些词作为动词出现时，几乎都是废话）
- 没对象："建议优化主页" → 错；"30 天内由市场部重拍 22 张主图" → 对
- 没时间窗：必须有"30 天""90 天""半年"等明确窗口
- 没预期：必须有"预期首页点击率提升 15-20%"或"具体效果需执行后复测"这种明确交代

### 平台术语红线

以下是 4 个最容易错、错了就会让付费客户怀疑整份报告真实性的点：

1. **携程的档次**：必须写"携程 X 钻 / 国标 Y 星（若挂牌）/ 平台档次 Z（豪华型/高档型/舒适型/经济型）"——三轨并存。只写"五星酒店"而忽略 98% 酒店没挂牌国标的事实，是错的
2. **美团的档次**：用"豪华型 / 高档型 / 舒适型 / 经济型"四档。**"钻石"在美团指会员等级（V1-V6），不是酒店档次，不要混用**
3. **"黑珍珠"是餐饮榜**（大众点评系），不是酒店榜。酒店榜叫**"必住榜"**。一家酒店如果有餐厅上了黑珍珠，可以作为差异化卖点，但不能写成"酒店黑珍珠"
4. **"必住榜"归属是大众点评**（不是美团），但由于美团和大众点评点评池打通，一家酒店在两个 App 上看到的"必住榜"标识一致。正确写法是"大众点评必住榜"，不是"美团必住榜"
5. **图片数量**：**一级页面首屏露出**（携程大约 7 张） vs **相册总数**（可能几十几百张），绝对不能混淆。写"主图仅 7 张"是典型的外行话，正确写法是"一级页面首屏露出 7 张，相册总数 X 张"

这些错误一次出现就足以让客户质疑整份报告，**写之前和提交前都要自检**。

## 语气参考（B 档：锋利结论风格）

- **像一位经验丰富的酒店行业顾问跟客户汇报**，不是像咨询公司 PPT 的客套话
- **保留"一句话起手式"式的犀利结论**（"这是一家被位置宠坏的老牌五星""一家产品比表达跑得更快的度假酒店"），写在报告总体结论和一页纸总结里
- **在行文允许使用判断色彩**——但必须在第二层，必须引用第一层事实
- **避免咨询行业黑话**（"赋能""闭环""抓手""底层逻辑""数字化转型""价值重塑"这类词一出现就错）

## 常见失败模式（看到自己在做时立刻停）

1. **没抓真实数据就开始写**——报告里全是"从主页来看"这种没有具体数字的句子
2. **小酒店硬套标准版**——稀释掉所有有用信息，反被客户看出外行
3. **用了"五星""四星"来描述没挂牌国标的酒店**——基本事实错误
4. **把大众点评的必住榜写成美团必住榜**，或把餐饮黑珍珠写成酒店黑珍珠
5. **竞品部分只列对手名字**，没给评分、点评数、装修年份、榜单荣誉的数字对比
6. **建议都是"优化""提升""改善"**——这是在写空话
7. **越层**：第一层写判断、第二层写事实陈列、第三层写空话
8. **生成 md 不转 PDF**——用户要的是 PDF 成品
9. **跳过 Step 0.5 平台可用性预检**——直接按用户选的 4 个平台抓，结果某个平台搜不到就默默略过或编造数据。**必须先预检、告知用户、再进正式抓取**
10. **酒店名有歧义时自作主张选一个**——比如用户说"花园酒店"，直接按广州花园酒店去写，结果用户要的是北京花园饭店。**歧义必须先问，不猜**

## 提交前自检清单

每份报告写完之后、转 PDF 之前，自己过一遍：

- [ ] 档次写法是否符合 `platform_facts.md`（三轨并存 / 四档分类 / 钻石不是档次）
- [ ] 图片数量是否区分了首屏露出 vs 相册总数

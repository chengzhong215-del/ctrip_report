# 报告章节骨架 · Report Template

## 使用说明

本文档提供**三种模板**：

1. **单平台标准版**（数据充足，16 分制 ≥12 分）—— 7 章完整结构
2. **单平台精简版**（数据中等，16 分制 6-11 分）—— 5 章精简结构
3. **多平台对比版**（用户明确要求横向对比）—— 6 章对比结构

严格按章节骨架写，但每个小节内容必须基于实际抓取数据，**禁止套用示例文字**。

所有模板都遵循 `analysis_framework.md` 的三层漏斗原则。

## 模板一：单平台标准版（7 章）

### 文件头部

```markdown

- **报告日期**：<YYYY-MM-DD>（由脚本自动填充当天日期）
- **数据口径**：<平台名> 主页 ✓ 已上架（hotel ID: <数字>，PC 端 + 移动端）
- **酒店属性**：<开业年份 / 装修年份 / 客房数>

- **集团归属**：<集团名 / 品牌定位，或"独立酒店">

```

### 一、总体评分与一句话结论

**一句话结论**（B 档锋利风格）：
格式："**这是一家 X 的酒店**"，要带判断色彩，不要中庸。参考 `analysis_framework.md` 末尾的起手式方法。

**五维等级表**：

| 维度 | 等级 | 一句话核心判断 |
|---|---|---|
| 平台表达完整度 | **B-（合格偏弱）** | 一句话判断 |
| 点评口碑健康度 | **B（合格）** | 一句话判断 |
| 产品差异化表达 | **B+（合格偏强）** | 一句话判断 |
| 商圈相对位置 | **A-（优秀）** | 一句话判断 |
| 流量转化效率 | **B-（合格偏弱）** | 一句话判断 |

打分等级：A / A- / B+ / B / B- / C+（不用 C 及以下，太难看且没信号）

### 三、诊断层：这家酒店的漏斗在哪里漏水

**本章的每个判断必须在同一段内引用上一章的具体数字。**

#### 3.1 转化漏斗诊断

按曝光 / 点击 / 浏览 / 下单四段，分别指出漏水阶段和证据。不一定每段都有问题，有问题的展开写。

#### 3.2 叙事缺口：资产与主页表达的错位

三小段结构：
- 它有什么资产（从 2.3 / 2.4 / 2.7 提取）
- 主页讲了什么（从 2.2 提取）
- 错位在哪，是浪费资产还是掩盖短板

#### 3.3 平台匹配度

这家酒店的产品特征 vs 所选平台的流量性格。参考 platform_facts.md 第五节。

结论分三档："天作之合 / 勉强搭配 / 平台错位"。

#### 3.4 点评运营健康度

- 回复率、差评归因结构、标签云与卖点的关系
- 有没有出现"老客再来"这类高价值点评
- 运营勤勉度的整体判断

### 四、杠杆层：可落地的优化动作

#### 4.1 高优（30-60 天，主页运营层）

4-5 条具体动作，每条四要素齐全：**做什么 × 谁做 × 何时做完 × 预期效果**

#### 4.2 中优（3-6 个月，产品/运营联动层）

3-4 条

#### 4.3 低优但长期关键（半年以上）

2-3 条

### 五、可追溯链自检表

| 建议（第四章） | 对应判断（第三章） | 判断依据（第二章） |
|---|---|---|
| ... | ... | ... |

**追不到 → 删**。

### 六、一页纸总结

blockquote 格式，B 档锋利风格：

```markdown
> **"一句起手式。"**
>
> 展开段落 1：核心资产或核心问题的结构。
>
> 展开段落 2：漏斗或叙事缺口的要害。
>
> 翻盘点：最有杠杆的一个动作 + 为什么它最值得做。
```

### 七、结尾附注

```markdown
*本报告基于<平台名>公开主页数据和同商圈竞品对比（数据截至 YYYY-MM-DD），未接触酒店后台运营数据。如需进一步做流量转化、ADR 对比或会员复购深度分析，需要酒店或平台侧提供后端数据配合。*
```

## 模板二：单平台精简版（5 章）

数据中等（6-11 分）时用。

### 文件头部

同标准版，但在"一、总体评分与结论"之前加一段：

```markdown
> **样本局限说明**：本报告基于<平台名>可公开抓取的主页数据。该酒店点评总数为 X 条，样本量相对有限，部分结论（尤其涉及差评归因、竞品对比、客群画像）的置信度低于大型酒店的同类报告。建议在点评积累到 500 条以上后再做标准版质检。
```

### 章节结构

1. **一、总体评分与结论**（打分维度减少到 3 个：平台表达 / 点评口碑 / 位置与竞品；不评"产品差异化表达"和"流量转化"）
2. **二、可观察层**（不拆小节，用一段流畅叙述 + 几张核心表）
3. **三、诊断层**（只写"叙事缺口" + "平台匹配度"两项，不展开漏斗和运营健康度）
4. **四、高优建议**（3 条以内，不分时间窗）
5. **五、一页纸总结**（保留，但段落更紧凑）

整体页数控制在 5-6 页。

### 三、平台覆盖说明（可选章节，仅当有未上架平台时出现）

**出现条件**：用户选的平台中存在未上架或状态异常的。如果所选平台全部已上架，**跳过本章**，后面的"诊断层"直接作为第三章。

对每个未上架平台写一小段说明，包含：

1. 未上架的客观事实
2. 该平台对这家酒店的**理论价值**（基于 platform_facts.md 第五节的客群画像判断）
3. 明确说明这是**渠道战略问题，不是主页运营问题**，不在本报告深度分析范围

示例写法：

> **关于飞猪平台**
>
> 该酒店目前未在飞猪上架主页。对一家国际品牌连锁酒店（凯悦/万豪/希尔顿等）而言，未接入飞猪意味着失去阿里生态的完整入口：5300 万 88VIP 用户（2025 年权益升级后享飞猪 F3 级福利）、芝麻信用住（近 10 万家酒店已接入）、会员通直升通道（凯悦若接入，飞猪高级会员可直升 Bonvoy 等集团会籍）。
>
> 这是集团层面的渠道战略选择，不是酒店主页运营问题，不在本报告深度分析范围。建议渠道团队评估是否补齐该入口。

**注意**：本章**不展开深度建议**。本 skill 只做单店主页质检，渠道战略建议超出范围。

### 四（或三）、诊断层：跨平台的结构性问题

#### 诊断 1：表达完整度差距

四个已上架平台中表达完整度最大的差距在哪？哪个平台几乎没做？

#### 诊断 2：流量分流诊断

- 哪个平台给了最大流量（点评数最多 / 榜单加成最高）
- 哪个平台有潜力但没发力（平台匹配度高但表达弱）
- 哪个平台应该减少投入（平台错位）

#### 诊断 3：跨平台的共性漏点 vs 平台独有漏点

- 共性：比如所有平台 C 位图都是通用房型，说明品牌级图片库有问题
- 独有：比如大众点评没展示酒店内黑珍珠餐厅，是大众点评独有的漏点

### 五（或四）、杠杆层：按平台分档的动作建议

**每个已上架平台独立给建议**，不混成一锅。未上架平台不在此展开（已在"平台覆盖说明"章处理）。

- **携程侧建议**：30 天内... / 3-6 个月...
- **美团侧建议**：30 天内... / 3-6 个月...
- **大众点评侧建议**：（如果有餐厅，餐厅侧和酒店侧的建议要分开）
- **飞猪侧建议**：（会员通、信用住、88VIP 权益接入是飞猪独有的发力点）
- **平台策略层建议**：主打哪 1-2 个平台、辅助哪几个、是否有必要全部做到同等水平

### 六（或五）、可追溯链自检表

同单平台标准版，但建议后要标注所属平台。

### 七（或六）、一页纸总结

同单平台标准版，但起手式要能概括"跨平台"的核心观察。

### 结尾附注

```markdown
*本报告基于 <实际抓取到的平台列表> 公开主页数据对比分析（数据截至 YYYY-MM-DD），未接触酒店后台运营数据。<如有未上架平台>该酒店未在 <平台名> 上架，相关渠道战略判断见第三章"平台覆盖说明"。<结束>多平台对比的结论依赖公开信息的时点一致性，平台主页信息存在时间性差异，建议重要决策前以最新抓取为准。*
```

## 起手式的几个参考句式（不要直接套词组）

- "这是一家被 X 宠坏的 Y"
- "一家产品比表达跑得更快的 X"
- "一家靠 X 吃饭、靠 Y 守分、但 Z 拖后腿的 W"
- "它有 A、B、C，但主页只给了 D、E"
- "[平台] 给了它 X，但它只兑现了 Y"
- （多平台对比）"在 A 被流量宠坏、在 B 被自己忽视、在 C 完全缺席"

起手式是整份报告的灵魂。**不要为了锋利而锋利**——它必须能回溯到第一层事实，否则就是哗众取宠，付费客户会立刻察觉。
